High-level project goal:
- Build a FastAPI backend for one user.
- Use Alpaca for broker access and options market data.
- Paper trading first; keep it live-trading-ready but do not optimize for live yet.
- Use Postgres as durable state.
- Keep the architecture simple, practical, and production-minded.

Core architecture decisions:
- Framework: FastAPI.
- Database: Postgres.
- ORM/migrations: SQLAlchemy 2.x and Alembic.
- Broker/data source: Alpaca.
- Hosting: Render.
- Auth: single admin bearer token from environment variables.
- Deployment style: free-first now, paid-ready later.
- Execution model: trades flow through previewed order_intents before execution.
- Source of truth for orders, fills, and positions: Alpaca.
- Execution and reconciliation run via Render cron workers.

- app/services/signal_scanner.py
  - Reads active strategies with config.scanner entries.
  - Scanner types: price_threshold, percent_change, moving_average, trend_confirmation.
  - moving_average: uses short/long window averages with bullish/bearish cross or trend triggers.
    Window shift of 1 bar is intentional to detect crossovers (documented in code).
    Requires long_window+1 bars to ensure enough data for the crossover comparison.
  - trend_confirmation: moving-average alignment plus minimum lookback percent change.
  - Supports scanner dedupe_minutes to suppress repeated equivalent signals.
  - Supports market regime check to block bullish signals in weak-market conditions.
  - Records job_runs and audit_logs for success/failure.
  - _average() is guarded against empty list to avoid ZeroDivisionError.

Current automation endpoints and their Render cron topology:
- Entry cycle (*/2 min, 9am-4pm ET): POST /api/v1/jobs/market-cycle
  - scan signals -> news check -> reconcile -> entry preview -> entry auto-submit
  - exit evaluation is permanently disabled on this route
- Exit cycle (*/1 min, 9am-4pm ET): POST /api/v1/jobs/market-cycle-exits
  - reconcile -> exit evaluation -> exit auto-submit
  - scan/preview/news are disabled; runs more frequently for faster exit protection
- Daily maintenance (8:30am + 5pm ET): POST /api/v1/jobs/market-maintenance
  - pre-market: reconcile, news, stale cleanup, readiness
  - post-market: deep reconcile, cleanup, performance summary

Current important limitations/gaps:
- Option contract selection is first-pass only; no delta/Greeks/moneyness/liquidity scoring yet.
- Exit evaluation supports P/L, expiration, and max_hold_hours rules; no trailing stops or AI-assisted exits yet.
- News scanning is lightweight RSS/headline gating; no persistent article table, sentiment model, article-body reading, or auto-sell decisioning yet.
- Statuses (order intent, broker order, signal) are plain strings, not a formal enum/state machine.
- No idempotency/retry protection beyond using order_intent id as Alpaca client_order_id.
- Reconciliation fetch limits are simple query params; no pagination loop yet.
- ai_trade_reviews and strategy_change_suggestions tables and ORM models exist (migration 0003). No automated review service is planned; review is manual.
- No local Docker Compose/Postgres helper yet.
- No integration tests against a real database yet.
- Render cron schedules should be reviewed periodically to confirm they match the entry/exit/maintenance split described above.

Trade case persistence:
- trade_cases, ai_trade_reviews, strategy_change_suggestions tables exist from migration 0003.
- ORM models exist in app/db/models.py.
- app/services/trade_cases.py: populate_trade_cases_from_closed_round_trips(db, limit=500)
  - Calls _fill_records + _match_round_trips from performance_review for FIFO matching.
  - Idempotency key: (entry_fill_id, exit_fill_id). Inserts new rows, updates context-changed rows, skips unchanged rows.
  - Stores context JSONB: {entry: {...}, exit: {...}, holding_seconds, entry_notional, exit_notional}.
  - Stores underlying_symbol by stripping the OCC option prefix (e.g. "SPY271219C00500000" → "SPY").
  - Route: POST /api/v1/jobs/populate-trade-cases?limit=500 (standalone, still available).
  - Automatically invoked at end of post-market maintenance (limit=5000, isolated transaction).
    Failure is caught by _populate_trade_cases_safely and reported under trade_cases.error without
    failing the maintenance job_run.
- GET /api/v1/automation/trade-cases provides a read-only FIFO trade-case view for manual review.
- ai_trade_reviews and strategy_change_suggestions tables exist for recording manual review notes.
  No automated review service is planned; no AI provider is called by the system.
