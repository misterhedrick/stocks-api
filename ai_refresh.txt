High-level project goal:
- Build a FastAPI backend for one user.
- Use Alpaca for broker access and options market data.
- Start with paper trading first.
- Keep the design live-trading-ready, but do not optimize for live trading yet.
- Use Postgres as durable state.
- Keep the architecture simple, practical, and production-minded.

Core architecture decisions:
- Framework: FastAPI.
- Database: Postgres.
- ORM/migrations: SQLAlchemy 2.x and Alembic.
- Broker/data source: Alpaca.
- Hosting: Render.
- Auth: single admin bearer token/API token from environment variables.
- Deployment style: free-first now, but paid-ready later.
- Execution model: trades should flow through previewed order_intents before execution.
- Source of truth for orders, fills, and positions: Alpaca.
- Longer term, execution and reconciliation should move to cron/worker-style jobs.

- Preview generation uses latest option quote only; no option-chain search/contract selection yet.

- No validation that strategy_id or signal_id exists when creating an order_intent.
- No formal status enum/state machine yet; statuses are plain strings.
- No idempotency/retry protection beyond using order_intent id as Alpaca client_order_id.
- Broker reconciliation exists only as a protected manual endpoint; it is not scheduled yet.

- No scheduled jobs are implemented yet.
